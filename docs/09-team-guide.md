## 🔄 Git 工作流

### 分支策略

```
main (生产分支)
  ├── develop (开发分支)
  │   ├── feature/user-management (功能分支)
  │   ├── feature/document-search (功能分支)
  │   └── feature/ai-chat (功能分支)
  ├── hotfix/critical-bug (紧急修复)
  └── release/v1.0.0 (发布分支)
```

### 分支命名规范

| 分支类型 | 命名格式 | 示例 |
|---------|---------|------|
| 功能开发 | feature/功能名 | feature/user-authentication |
| Bug修复 | bug/问题描述 | bug/login-error |
| 紧急修复 | hotfix/问题描述 | hotfix/database-connection |
| 发布版本 | release/版本号 | release/v1.0.0 |

### Git 提交规范

#### Commit Message 格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

#### Type 类型

| 类型 | 说明 | 示例 |
|------|------|------|
| feat | 新功能 | feat(auth): add JWT authentication |
| fix | Bug修复 | fix(document): resolve null pointer exception |
| docs | 文档更新 | docs(readme): update installation guide |
| style | 代码格式 | style(api): format code with prettier |
| refactor | 重构 | refactor(service): simplify user service |
| test | 测试 | test(controller): add unit tests |
| chore | 构建/工具 | chore(deps): upgrade dependencies |

#### 示例

```bash
# 好的提交
git commit -m "feat(auth): add JWT token refresh mechanism

- Implement token refresh endpoint
- Add refresh token validation
- Update authentication middleware

Closes #123"

# 不好的提交
git commit -m "update code"
git commit -m "fix bug"
```

## 📝 代码审查流程

### Pull Request 流程

1. **创建功能分支**
   ```bash
   git checkout -b feature/new-feature
   ```

2. **开发并提交**
   ```bash
   git add .
   git commit -m "feat: add new feature"
   ```

3. **推送到远程**
   ```bash
   git push origin feature/new-feature
   ```

4. **创建 Pull Request**
   - 标题：简短描述
   - 描述：详细说明和截图
   - 关联 Issue：#123

5. **代码审查**
   - 至少 1 人审查
   - 检查代码质量
   - 验证功能

6. **合并代码**
   - Squash 合并
   - 删除功能分支

### 代码审查清单

#### 代码质量
- [ ] 代码符合项目规范
- [ ] 变量和函数命名清晰
- [ ] 没有冗余代码
- [ ] 错误处理完善

#### 功能实现
- [ ] 符合需求文档
- [ ] 边界情况处理
- [ ] 性能考虑
- [ ] 安全检查

#### 测试覆盖
- [ ] 单元测试完整
- [ ] 集成测试通过
- [ ] 手动测试验证

### Bug 优先级

| 优先级 | 响应时间 | 修复时间 | 示例 |
|--------|---------|---------|------|
| P0 - 紧急 | 1小时 | 24小时 | 系统崩溃、数据丢失 |
| P1 - 高 | 4小时 | 3天 | 核心功能不可用 |
| P2 - 中 | 1天 | 1周 | 功能异常但有替代方案 |
| P3 - 低 | 3天 | 2周 | UI问题、优化建议 |

## 🔄 发布流程

### 版本号规范

```
major.minor.patch

例如: 1.2.3
- major: 重大功能变更或不兼容更新
- minor: 新功能但向后兼容
- patch: Bug修复
```

### 发布检查清单

#### 发布前
- [ ] 所有测试通过
- [ ] 代码审查完成
- [ ] 文档更新完成
- [ ] 数据库迁移脚本准备
- [ ] 回滚方案准备

#### 发布中
- [ ] 备份数据库
- [ ] 执行数据库迁移
- [ ] 部署新版本
- [ ] 验证核心功能
- [ ] 监控系统日志

#### 发布后
- [ ] 通知相关人员
- [ ] 更新文档
- [ ] 监控系统状态
- [ ] 收集用户反馈

## 🔐 安全规范

### 开发安全

#### 密码管理
- 不在代码中硬编码密码
- 使用环境变量存储密钥
- 定期更换密钥

#### 依赖安全
- 定期更新依赖包
- 使用安全扫描工具
- 及时修复安全漏洞

#### 数据安全
- 敏感数据加密存储
- API接口权限控制
- 日志脱敏处理
